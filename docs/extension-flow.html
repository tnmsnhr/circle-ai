Chrome Extension 101

A Chrome extension is not a single web app. It is a small package Chrome loads from manifest.json, made of separate JavaScript programs that run in different contexts and talk through Chrome APIs.

Manifest V3 (MV3) is the current format. Syncle uses MV3. The “background page” from old extensions is now a service worker (background.js) — it can sleep, wake on events, and must not assume it stays alive forever.

What happens when a user installs Syncle

  1. 1
    Chrome reads manifest.json
    Declares scripts, permissions, popup, host access
  2. 2
    Background service worker starts
    Handles fetch proxy + tab screenshots
  3. 3
    Content script injects on web pages
    Runs OverlayApp in Shadow DOM
  4. 4
    Popup opens on icon click
    Settings, sign-in — does not draw lassos

Three JavaScript worlds

These cannot call each other’s variables directly. They communicate with chrome.runtime.sendMessage, chrome.storage, or shared DOM (content script only).

Page (host site)Yes (its own)NoUntrusted — Syncle does not run code here
Content scriptYes (inject UI)Partial (tabs, runtime, storage)Lasso, bubbles, calling background
Service workerNoYesFetch to API, captureVisibleTab
Extension popupNoYesSettings, Google sign-in

Why Shadow DOM? Content script CSS would fight the host page. Syncle mounts React inside #draw-on-web-root-host shadow root so lasso styles stay predictable. Popups/toolbar portal to light-DOM mounts so pointer-events work correctly.

Dev workflow

  1. Write source in src/ and public/
  2. Run npm run build → output in dist/
  3. Chrome → Extensions → Load unpacked → select dist/
  4. After code changes: rebuild, then click Reload on the extension card

MV3 pieces — what each part does in Syncle

manifest.json

The contract Chrome enforces. If it’s not declared here, your code cannot do it.

FieldSyncle valueMeaning
content_scriptscontent-script.js @ document_endAuto-inject overlay on every page load
backgroundbackground.jsService worker — no DOM, event-driven
action.popuppopup.htmlSettings panel from toolbar icon
permissions.offscreen—Allows hidden offscreen.html for image processing
permissions.tabs—captureVisibleTab for screenshots
host_permissions<all_urls> + API hostInject + call syncle-services

background.js (service worker)

Runs in the extension origin, not on the webpage. Wakes up, handles messages, may sleep again.

Message typeWhat it does
SYNCLE_FETCHProxies HTTP to syncle-services (avoids page CORS / carries auth from extension)
EXTRACTION_CAPTURE_CROPcaptureVisibleTab → send PNG to offscreen → return JPEG base64 crop
PINGHealth check

content-script.js (built from src/)

The “app on the page.” Bundled by Vite as one IIFE file (MV3 disallows multi-chunk content scripts easily).

  • Entry: src/content-script.jsx → mounts OverlayApp
  • Can read/modify DOM (lasso canvases, portals)
  • Cannot access page JS variables (isolated world)
  • Sends messages to background for fetch + screenshots

popup.html + PopupApp

Small UI when you click the Syncle icon. Separate React app (second Vite build).

  • Enable/disable drawing, lasso theme, auto-collapse
  • Google sign-in → stores JWT in chrome.storage.local
  • Writes chrome.storage.sync → content script listens via chrome.storage.onChanged

offscreen.html + offscreen.js

MV3 service workers cannot use DOM APIs like canvas or Image. Chrome provides a hidden document for that.

  • Background captures full-tab PNG
  • Offscreen loads image, crops to selection rect, encodes JPEG
  • Returns base64 to content script via message chain

Messaging pattern (how worlds talk)

Syncle overview — end-to-end flow

Syncle is a Chrome MV3 extension. On every page it injects a content script that mounts OverlayApp in a Shadow DOM. Users hold ⌘/Ctrl, draw a lasso, get a screenshot crop of the region, register context with syncle-services, and receive an AI explanation in a pinned popup bubble.
